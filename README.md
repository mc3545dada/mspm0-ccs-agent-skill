# MSPM0 CCS Agent Skill

A CLI-agent rules package for TI MSPM0 development with Code Composer Studio, SysConfig, and DriverLib.

Generic coding agents often understand C syntax but miss the workflow rules around TI CCS projects. The common failure mode is editing generated files, guessing pinmux code, or using the wrong generated macro names. This repository teaches agents how to work with MSPM0 projects safely.

Primary targets:

- Claude Code
- OpenCode
- OpenClaw
- Continue
- Cursor
- Other CLI or editor-based coding agents that read `SKILL.md`, `AGENTS.md`, or `CLAUDE.md`

Codex and other skill-aware tools can also use the same files, but this project is not Codex-specific.

## Why This Exists

MSPM0 CCS projects often use SysConfig. The `.syscfg` file is the editable source of truth for pins, peripherals, clocks, and generated initialization code. Files such as `ti_msp_dl_config.c` and `ti_msp_dl_config.h` are generated outputs and should not be patched by hand.

This matters for:

- MSPM0G3507 and MSPM0L-series projects
- TI LaunchPad examples
- LCKFB Tianmengxing MSPM0G3507 projects
- NUEDC student projects
- Motor, servo, UART, OLED, ADC, timer, and PID/control workflows

## Quick Start

Copy one or more entry files into an MSPM0 CCS project, depending on your agent:

```text
SKILL.md
AGENTS.md
CLAUDE.md
```

Then ask the agent to inspect the project before editing:

```text
Read the MSPM0 agent rules, inspect the .syscfg file and generated ti_msp_dl_config.h, then help me add a UART debug interface safely.
```

After a `.syscfg` change, rebuild the CCS project or run the SysConfig CLI command generated by CCS.

## Current Scope

Phase 1 provides documentation and agent rules only:

- `SKILL.md`: tool-neutral skill entry point for skill-aware CLI agents
- `AGENTS.md`: broad cross-agent rules
- `CLAUDE.md`: Claude Code-oriented instructions
- `docs/syscfg_rules.md`: safe `.syscfg` editing rules
- `docs/ccs_project_rules.md`: CCS project layout rules
- `docs/driverlib_rules.md`: DriverLib usage rules
- `docs/common_mistakes.md`: mistakes agents should avoid

No physical MSPM0 board is required for Phase 1. The focus is source inspection and build-time validation.

## Key Rules

- Treat `.syscfg` as the source of truth for pinmux and peripheral configuration.
- Do not manually edit generated SysConfig outputs such as `Debug/ti_msp_dl_config.c` or `Debug/ti_msp_dl_config.h`.
- Preserve SysConfig metadata, including device, package, SDK product, and tool version comments.
- Prefer DriverLib APIs over hand-written register code unless the user explicitly asks otherwise.
- Read the generated header to confirm macro names and init function spelling.
- Rebuild or run SysConfig after modifying `.syscfg`.
- Ask before migrating device, package, board, SDK, compiler, or CCS version.
