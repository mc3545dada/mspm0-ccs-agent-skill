# SysConfig Schema Sources

There is no single official MSPM0 `.syscfg` field manual that lists every module field, enum value, solver rule, and clock-tree option in a human-oriented syntax reference.

For agent work, treat `.syscfg` editing as evidence-based modification, not freehand authoring.

## Source Priority

When adding or changing SysConfig modules, use sources in this order:

1. The user's existing `.syscfg` that already opens in SysConfig or builds in CCS.
2. MSPM0 SDK example `.syscfg` files under `examples/`.
3. SysConfig GUI or standalone-generated `untitled.syscfg` for the same device, package, board, SDK, and tool version.
4. Skill-provided scaffolds under `assets/snippets/`.

Do not invent device, package, product, board, or version metadata from memory.

## Where Fields Come From

General `.syscfg` script patterns can be learned from examples:

```js
const GPIO = scripting.addModule("/ti/driverlib/GPIO", {}, false);
const GPIO1 = GPIO.addInstance();

GPIO1.$name = "LED";
GPIO1.port = "PORTB";
GPIO1.associatedPins[0].assignedPin = "22";
```

This shows how to add modules, add instances, and assign fields. It does not prove which fields and enum values are valid for every module.

For module-specific field names, defaults, enum options, and validation logic, inspect the local MSPM0 SDK metadata:

```text
<mspm0_sdk>/source/ti/driverlib/.meta/*.syscfg.js
```

Common files include:

```text
GPIO.syscfg.js
UART.syscfg.js
SYSCTL.syscfg.js
PWM.syscfg.js
TIMER.syscfg.js
ADC12.syscfg.js
I2C.syscfg.js
SPI.syscfg.js
DMA.syscfg.js
```

These files are SysConfig module definitions, not user project configuration files. Read them to confirm field names and available options, then validate by running SysConfig CLI or a CCS build.

## SDK Examples

MSPM0 SDK examples are the most reliable few-shot source for complete working configuration patterns:

```text
<mspm0_sdk>/examples/**/*.syscfg
```

Prefer examples with the same board, device family, SDK version, and peripheral. For Tianmengxing MSPM0G3507 work, TI `LP_MSPM0G3507` examples are usually a better starting point than unrelated MSPM0L examples, but board pin maps still need Tianmengxing verification.

Use `scripts/index_syscfg_examples.py` to find examples and metadata quickly:

```powershell
python skills\mspm0-ccs\scripts\index_syscfg_examples.py C:\ti\mspm0_sdk_2_10_00_04 --board LP_MSPM0G3507 --module UART
```

## Metadata Rules

Preserve `.syscfg` metadata:

```text
@cliArgs
@v2CliArgs
@versions
--device
--package
--product
```

If these values are missing or suspicious, do not guess. Ask the user to create or open a matching configuration in SysConfig GUI or standalone SysConfig, save it, and provide the generated `.syscfg` header.

For MSPM0G3507 LQFP-64 with MSPM0 SDK 2.10.00.04, see:

```text
assets/snippets/mspm0g3507_lqfp64_empty_scaffold.syscfg.md
```

Use that only as a known-good local scaffold, not as proof that another SDK, package, or board uses the same metadata.

## Validation

After modifying `.syscfg`, run SysConfig CLI or rebuild the CCS project.

Typical inputs:

```text
--script <project>.syscfg
-o <output-directory>
-s <mspm0_sdk>/.metadata/product.json
--compiler ticlang
```

Prefer the command generated by CCS in `Debug/subdir_rules.mk` when it exists. If validation cannot be run locally, explicitly report: `SysConfig validation was not completed`.
